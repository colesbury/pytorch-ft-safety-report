# Free-Threading Safety Audit: lazy_generated

## Overall Risk: NONE

## Files Audited
- `lazy/generated/LazyIr.h`
- `lazy/generated/LazyNativeFunctions.cpp`
- `lazy/generated/LazyNativeFunctions.h`
- `lazy/generated/LazyNonNativeIr.h`
- `lazy/generated/RegisterAutogradLazy.cpp`
- `lazy/generated/RegisterLazy.cpp`

## Findings

No thread-safety issues found.

### Analysis

These files are auto-generated by `tools/setup_helpers/generate_code.py`.

**LazyIr.h:** Defines generated IR node classes (e.g., `AdaptiveAvgPool2d`, `Convolution`, etc.) that extend `TsNode`. These are pure data classes with constructors, `ToString()`, `CanBeReused()`, and `Lower()` methods. No static/global mutable state. The `static const torch::lazy::Value kNullValue` is a const object.

**LazyNativeFunctions.cpp:** Contains generated implementations of lazy native functions. Each function is a dispatch entry point that:
1. Checks `force_eager_fallback()` (reads a flag -- see ts_eager_fallback analysis)
2. Creates IR nodes
3. Returns lazy tensors

These functions contain no static/global mutable state themselves. The `to_meta()` helper functions at namespace scope are stateless.

**RegisterAutogradLazy.cpp / RegisterLazy.cpp:** Generated dispatch registration files. They use `TORCH_LIBRARY_IMPL` macros which register operator implementations at static init time. No mutable global state after init.

## Summary

All generated files are either stateless function implementations, pure data class definitions, or dispatch registrations that execute at static init time. No Python C API usage, no mutable global/static state, no thread-safety concerns. The generated code delegates all stateful operations to the core lazy tensor infrastructure which has its own synchronization.
